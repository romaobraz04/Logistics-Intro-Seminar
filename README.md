# Battery Energy Storage Optimization

This project optimizes the operation of a lithium-ion battery to maximize profit through **energy arbitrage** — charging when electricity prices are low and discharging when they are high. The battery must respect physical constraints (capacity limits, power limits, charging efficiency) and optional economic penalties (grid fees, degradation costs).

Two competing Mixed-Integer Programming (MIP) formulations are implemented and compared across a series of assignments, each exploring a different aspect of the problem:

| Assignment | Question answered |
|---|---|
| 4 | How does formulation quality (basic vs. tighter) affect solver speed and LP relaxation gaps? |
| 5 | How do grid import/export fees change battery behavior and profitability? |
| 6 | How does a linear battery degradation cost affect throughput and cycling? |
| 7 | How much profit is lost when the battery operator has limited price foresight (rolling horizon)? |

## Configuration Reference

The main config file is `code/configs/base_case.json`. Assignment configs add scenario-specific parameters on top of this.

```json
{
  "name": "case1_battery_optimization",
  "data": {
    "csv_path": "../../net_demand_and_price.csv",
    "start_index": 0,
    "periods": null
  },
  "battery": {
    "initial_soc_kwh": 2.0,
    "soc_min_kwh": 0.5,
    "soc_max_kwh": 5.0,
    "charge_power_limit_kw": 2.0,
    "discharge_power_limit_kw": 2.0,
    "charge_efficiency": 0.95,
    "discharge_efficiency": 0.95,
    "grid_fee_eur_per_kwh": 0.0,
    "degradation_cost_eur_per_kwh_throughput": 0.0
  },
  "solver": {
    "time_limit_seconds": 600,
    "mip_gap": 0.0,
    "threads": 0,
    "output_flag": 0
  }
}
```

### `data` fields

| Field | Type | Description |
|---|---|---|
| `csv_path` | string | Path to the dataset CSV, resolved relative to the config file (not your working directory) |
| `start_index` | int | Row index of the first time step to include (0 = beginning of dataset) |
| `periods` | int or null | Number of hourly periods to solve; `null` uses the full dataset |

### `battery` fields

| Field | Units | Description |
|---|---|---|
| `initial_soc_kwh` | kWh | State of charge at the start of the planning horizon |
| `soc_min_kwh` | kWh | Minimum allowed state of charge (depth-of-discharge limit) |
| `soc_max_kwh` | kWh | Maximum capacity (fully charged) |
| `charge_power_limit_kw` | kW | Maximum charging rate |
| `discharge_power_limit_kw` | kW | Maximum discharging rate |
| `charge_efficiency` | 0–1 | Fraction of drawn grid power that enters the battery |
| `discharge_efficiency` | 0–1 | Fraction of battery energy that reaches the grid |
| `grid_fee_eur_per_kwh` | EUR/kWh | Fee applied to every kWh imported or exported; set to 0 to disable |
| `degradation_cost_eur_per_kwh_throughput` | EUR/kWh | Linear cost per kWh cycled through the battery; set to 0 to disable |

### `solver` fields

| Field | Description |
|---|---|
| `time_limit_seconds` | Maximum solver wall-clock time; increase if the solver hits the limit before finding an optimal solution |
| `mip_gap` | Relative MIP optimality gap tolerance (0.0 = prove optimality; 0.01 = stop at 1% gap) |
| `threads` | Number of solver threads (0 = use all available cores) |

## Running the Scripts

All scripts are run from the **repository root** (the folder containing this README). They automatically locate configs and the dataset using paths relative to the config files.

### Base case

Solves both formulations and prints a side-by-side comparison:

```bash
python code/scripts/run_base_case.py
```

### Assignment 4 — Formulation benchmark and LP relaxation study

```bash
python code/scripts/run_assignment4.py
```

Produces under `code/outputs/assignment4/`:
- `benchmark.csv` — solve time and objective for basic vs. tighter on the full horizon
- `horizon_scaling.csv` — objective and runtime across fractional data windows (10 %–100 %)
- `lp_gap.csv` — LP relaxation gaps for various parameter scenarios (price spread, efficiency, power limits, capacity)
- Corresponding PNG plots

### Assignment 5 — Grid-fee sweep

```bash
python code/scripts/run_assignment5.py
```

Produces under `code/outputs/assignment5/`:
- `grid_fee_sweep.csv` — profit, grid import/export volumes, and self-consumption proxy across fee levels
- Corresponding PNG plots

### Assignment 6 — Degradation-cost sweep

```bash
python code/scripts/run_assignment6.py
```

Produces under `code/outputs/assignment6/`:
- `degradation_sweep.csv` — objective, battery throughput, and equivalent cycles vs. degradation cost
- Corresponding PNG plots

### Assignment 7 — Rolling horizon (limited foresight)

```bash
python code/scripts/run_assignment7.py
```

Produces under `code/outputs/assignment7/`:
- `rolling_horizon.csv` — profit under perfect foresight vs. weekly / 2-day / day-ahead windows
- Value of perfect information (VPI) figures

## CLI Usage

If you prefer to call the package directly (e.g., for custom configs or one-off runs), use the CLI from inside the `code/` directory:

```bash
cd code

# Solve a single formulation
python -m battery_opt.cli solve --config configs/base_case.json --formulation basic

# Solve with LP relaxation (continuous binaries)
python -m battery_opt.cli solve --config configs/base_case.json --formulation tighter --relax-binaries

# Benchmark both formulations, save result table
python -m battery_opt.cli benchmark --config configs/base_case.json --output outputs/benchmark.csv

# Run full assignment analyses
python -m battery_opt.cli assignment4 --config configs/base_case.json --analysis-config configs/assignment4.json --output-dir outputs/assignment4
python -m battery_opt.cli assignment5 --config configs/base_case.json --analysis-config configs/assignment5.json --output-dir outputs/assignment5

```

## Notes and Troubleshooting

**Config paths are resolved relative to the config file, not your working directory.**
The scripts work correctly whether you run them from the repository root or from inside `code/`. Do not change `csv_path` unless you move the dataset file.

**Solver hits the time limit.**
If you see `status=NOT_SOLVED` or a large MIP gap, increase `solver.time_limit_seconds` in `base_case.json`, or reduce `data.periods` to use a shorter planning horizon.

**Assignment 5 uses a self-consumption proxy.**
The dataset only provides net demand (load minus PV generation), not separate load and PV time series. The self-consumption metric is therefore an approximation.

**Assignment 6 uses a linear degradation model.**
The cost is proportional to total kWh cycled through the battery (charge + discharge). This keeps the model linear and easy to interpret. More accurate degradation models (e.g., rainflow counting) would require a nonlinear formulation.

**Assignment 7 rolling horizon scenarios.**
The perfect-foresight result serves as an upper bound. Day-ahead and shorter windows incur a value-of-information loss because the optimizer cannot anticipate future prices.
